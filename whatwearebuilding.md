# 生鮮團購訂購系統 — MVP Spec

小型生鮮團購的訂購 + 金流回報 + 出貨管理 + 供應商管理系統。
團主在 LINE 群組分享連結 → 用戶下單 + 匯款回報 → 團主確認付款 → 協調供應商到貨 → 確認出貨 + 通知。
內建類募資進度條，鼓勵揪團達標。

## 核心概念

- 取代 LINE 群組接龍 + Excel 手動記帳
- 用戶零註冊（可選擇儲存 `暱稱 + 電話` 對應的結帳資料，供下次自動帶入）
- 團購進度條（類 Kickstarter）驅動分享裂變
- Admin-only 登入，用戶端完全公開
- 完整訂單生命週期：下單 → 付款 → 確認 → 到貨通知 → 出貨
- LINE 綁定推播：用戶貼 `訂單編號 + 訂購人姓名 + 手機末三碼` 到 LINE OA → 1-on-1 push
- Admin POS：代客下單、現場收款快速確認、列印出貨單、面交查單交貨
- 單一開團：系統同一時間只允許一個 open round（DB partial unique index 強制）
- 視覺語言：溫潤、自然、精品市場感，不走工具站或科技感 dashboard 風格
- 公開端與後台共用同一品牌系統：暖白紙感底、深橄欖字色、森林綠主按鈕、青銅色輔助標籤、編輯感襯線標題

> **Technical details** (schema, status flow, RLS, API contracts, terminology) are in `claude.md`. This file covers product spec and UX only.

## User Flows

### Flow 1：用戶瀏覽 + 下單

1. 用戶點連結 → 商品列表，每個商品顯示品名、單價、單位、庫存、進度條（已被預訂/目標或庫存）
2. +/- 加入購物車（庫存上限 disabled）
3. 底部浮動 CartBar 顯示件數 + 金額 + 宅配運費提示 → 點「下一步」
4. 填寫：`暱稱 / 訂購人 / 收貨人`、電話、地址、Email、取貨方式（宅配顯示運費；面交選項依本輪設定的面交點 A/B labels）、訂單摘要
5. 只有當 `暱稱 + 完整電話` 都填好時，才嘗試自動帶入已儲存資料；目前「完整電話」定義為 normalize 後至少 10 碼，未達門檻時不打 lookup、也不消耗 lookup rate limit。帶入內容只含 `訂購人 / 收貨人 / 地址 / Email`
6. 可勾選 `儲存資料，下次結帳自動帶入`
7. 若勾選儲存且同暱稱已有資料，但完整電話不同 → 不允許覆寫已儲存資料；可取消勾選後直接下單，或改用其他暱稱
8. 送出 → 按鈕 disabled + `submission_key` 防重複 → server 驗庫存 → 計運費 → 原子扣庫存 → 建訂單

### Flow 2：匯款 + 回報

1. 訂單成立頁：銀行帳號、訂單編號、分享揪團 CTA（商品未達標時）、取消按鈕（pending_payment）
2. 匯款後 → 回報（金額 + 帳號後五碼）→ 確認步驟（金額對比）→ 送出
3. 狀態：`pending_payment` → `pending_confirm`

### Flow 2.5：LINE 綁定推播

1. 訂單頁提供可複製的 `ORD-YYYYMMDD-NNN 王小美 678` 綁定內容，並提供直達 LINE OA 按鈕
2. Webhook 驗證訂單編號 + 訂購人姓名 + 手機末三碼 → 綁定 `line_user_id`
3. 後續通知走 1-on-1 push。已綁定再傳 → 顯示狀態（冪等）。格式錯誤 → 使用說明
4. 每訂單只能綁一個 LINE 帳號。未綁定 → 跳過 push，僅發 Email

### Flow 3：訂單查詢

1. `/lookup` 輸入訂購人姓名 + 手機末三碼 → 顯示匹配訂單（新到舊）
2. 點入單筆 → 在同一瀏覽器 session 內直接開啟詳情，不再重複要求同一組姓名與手機末三碼
3. 若是直接開 `/order/[orderNumber]` 而非從查詢頁進入，仍需輸入訂購人姓名 + 手機末三碼解鎖
4. 解鎖後顯示收據式訂單摘要、聯絡電話、收貨地址（若有）、取貨方式、LINE 綁定資訊
5. `pending_payment` 可操作：匯款回報 / 取消

### Flow 4：Admin 確認付款

1. 儀表板：摘要（總訂單、營收、各狀態數量）+ 商品需求彙總（供貨商用）+ 通知狀態
2. 訂單管理：篩選 tabs + 搜尋 + 批次確認 + 單筆詳情（匯款比對 → 確認/取消）+ CSV
3. 確認 → `confirmed` + `confirmed_at` → LINE + Email 通知

### Flow 5：Admin 出貨管理

1. 待出貨頁列 `confirmed` 訂單，按取貨方式分組（宅配/面交點）
2. 搜尋 + 單筆確認（"確認寄出"/"確認取貨"）+ 批次確認
3. → `shipped` + `shipped_at` → LINE + Email 出貨通知

### Flow 6：Admin 供應商管理

1. 供應商 CRUD（名稱、聯絡人、電話、Email、商品數）
2. 商品關聯供應商。點商品展開客戶列表（暱稱、訂購人、收貨人、電話、數量、訂單編號）
3. 「通知到貨」→ 通知訂購該商品的所有客戶（LINE multicast + Email）；客戶數以邏輯訂購人身份（`訂購人 + 電話`）去重，不以暱稱當身份

### Flow 7：Admin POS

1. 「+ 代客下單」→ 選商品 + 填客戶資料（Admin 以暱稱自動帶入已儲存資料，若沒有則 fallback 最新訂單快照）
2. 送出 → `pending_payment`
3. 可立即「已現場收款」→ 直接 `confirmed`（quick-confirm）

### Flow 8：Admin 取消

- 任何狀態皆可取消（附原因）。恢復庫存（shipped 除外）。發取消通知

### Flow 9：面交 POS

- 篩選面交 → 搜尋客戶 → 確認取貨。未付款先收款再取貨

### Flow 10：列印

- 訂單詳情 → 列印出貨單（@media print）
- 待出貨頁 → 批次列印
- 商品需求彙總 → 列印理貨清單（商品名 + 客戶分配）

### Flow 11：商品 + 開團管理

- 商品：CRUD + 上下架 + 庫存 + 目標量 + 圖片 + 關聯供應商
- 開團：新開（名稱 + 截止 + 運費 + 面交點 A/B）/ 截單 / 修改。截單後用戶端禁止下單
- 歷史團次顯示最近 5 團

## 進度條機制

- Admin 設 `goal_qty` → `product_progress` view 即時算 `current_qty`（排除取消）
- 無庫存上限時：維持原本 goal-only 進度條；未達標顯示橘色，已達標顯示綠色
- 有有限庫存時：進度條改以「庫存上限」為整條長度，`成團目標` 只作為條上的標記，不因達標就整條填滿
- 已達標但仍有庫存時：進度條仍保留可見餘量，避免誤解成已賣完
- 公開商品卡文案要明確表達這是全團累計量，不是單一客戶購物車：`已有 x被預訂，共有 y`
- 無目標 → 不顯示
- 加入購物車時 optimistic 更新：`current_qty + cart_qty`

## 分享揪團 CTA

任一商品未達標時顯示（商品頁底部、訂單成立頁、匯款回報完成頁）：
- 「🔥 幫忙揪團，一起達標！」
- 「📋 複製連結」+ 「💬 分享到 LINE」+ 完整連結

## 運費機制

- Admin 開團時設 `shipping_fee`。可修改（不影響已成立訂單）。null/0 = 免運
- 用戶端首頁摘要：顯示 `本團運費 {n}元`；若 round 尚未設定則顯示 `本團運費待設定`。另獨立顯示 `面交取貨免運`
- 用戶端 checkout：選宅配 → 顯示「宅配到以下地址，另收運費」。面交 → 不顯示
- API：宅配 → `order.shipping_fee = round.shipping_fee`，面交 → null（快照原則）

## 面交點機制

- 每一團都有自己的 `pickup_option_a` / `pickup_option_b`，預設為「面交點 A / 面交點 B」
- Admin 可在開團管理頁修改當前開團的面交點名稱，也可在建立新團時直接設定
- 公開結帳頁與 Admin POS 都必須即時跟隨當前 round 的面交點設定，不可寫死在前端常數
- API / DB 驗證以 round 的實際設定為準；`pickup_location` 只允許空字串（宅配）或該 round 的 A/B label

## UX 要點

### 用戶端

1. CartBar 為深色浮動托盤，主金額高對比，次要資訊仍需維持可讀性
2. 截單即時檢查：結帳頁 mount 時重檢 `round.is_open`
3. 匯款回報確認步驟：金額 + 後五碼 → 確認摘要 → 送出
4. 訂單查詢可點進詳情
5. 訂單成立頁「繼續選購」按鈕
6. 商品卡優先使用圖片；無圖時仍需有精緻的 placeholder，而不是空白區塊
7. Lookup / 解鎖 / 訂單詳情應像精品收據或會員驗證面板，不是單純表單堆疊
8. 經 `/lookup` 驗證成功後，點入單筆訂單應直接開啟詳情，不要再次要求輸入相同資訊
9. LINE 綁定卡必須提供低摩擦操作：可直接複製綁定字串，也可直接開啟官方 LINE OA
10. 解鎖後的單筆訂單頁要顯示收件聯絡電話與收貨地址，避免用戶回頭查聊天記錄
11. 商品卡上的剩餘庫存提示要更醒目、對比更高，讓中高齡使用者能快速感受到「快沒了」
12. 商品卡圖面標題區要維持視覺置中，不要掉到圖片最底部
13. round window 狀態 pill 的 `開放中` 不可自動換行成兩列

### Admin 端

1. 儀表板數字可點擊跳轉（待確認 → 訂單頁、待出貨 → 出貨頁）
2. 訂單頁搜尋欄（暱稱/訂購人/收貨人/電話/訂單編號）即時搜尋
3. 待出貨頁按取貨方式分組顯示
4. 通知到貨後顯示「前往出貨」跳轉
5. 固定導航列 + 登出 + 當前團次名稱
6. 響應式：平板可用，按鈕 ≥ 44px
7. 後台雖然要有效率，但視覺上仍需和公開端同一家族，避免回退成通用白卡片 + 亮色 tab 的工具感

- 已完成一輪大型視覺重構：
  - 公開首頁：編輯感 hero、圖片優先商品卡、浮動購物托盤、分段式 checkout
  - 訂單查詢 / 單筆訂單：高質感驗證面板、收據式摘要、較完整的匯款與 LINE 綁定呈現
  - 後台：統一品牌色與字體、重做導航、儀表板卡片、訂單卡、出貨卡、商品/供應商/開團頁與表單
- 已補上字體載入，避免只依賴系統 fallback：
  - `@fontsource/noto-sans-tc`
  - `@fontsource/noto-serif-tc`
- 最近補強的流程細節：
  - 訂單送出後與 `/lookup` 點入結果時，會走 signed detail URL → httpOnly cookie 的解鎖流程，`/order/[orderNumber]` 可直接 server-render，不再依賴 client-side 暫存解鎖資料
  - 公開訂單頁的 LINE 綁定區塊已加入「一鍵複製綁定內容」與直達官方 LINE OA 按鈕
  - 解鎖後的公開訂單頁會顯示聯絡電話與收貨地址
  - 面交點 A/B 已改為 round-level 設定，公開 checkout 與 admin POS 會同步反映
  - 有限庫存商品的進度條已改成以庫存上限為整條長度，`成團目標` 只作標記；達標不再等於視覺上「整條滿格」
  - 商品卡的 `剩餘` badge 已放大並提升對比，降低被忽略的機率
  - `/lookup` 驗證後收到的是可直接開啟詳情的 signed detail links，不再在單筆訂單頁重複驗證；只有直接進入 `/order/[orderNumber]` 時才顯示手動驗證表單
  - 查詢結果 CTA 已加上中文文案 `查詢細節`，避免只顯示英文 `view detail`
  - 公開 checkout 已改成 `暱稱 / 訂購人 / 收貨人` 三欄位，並加入 opt-in 的 `儲存資料，下次結帳自動帶入`
  - 公開 autofill 只有在 `暱稱 + 完整電話` 同時存在時才會觸發，資料源是 `saved_checkout_profiles`；完整電話門檻為 normalize 後至少 10 碼，未達門檻時不查詢也不顯示 phone mismatch
  - 公開驗證流程（lookup / 單筆訂單解鎖 / 匯款回報 / 取消 / LINE 綁定）已統一改成使用 `訂購人姓名 + 手機末三碼`
  - admin 訂單 / 出貨 / 商品彙總 / 供應商客戶清單 / CSV / 出貨單都會顯示 `暱稱 / 訂購人 / 收貨人`
  - 到貨通知/供應商統計中的客戶數已改成依 `訂購人 + 電話` 邏輯身份去重，不再把暱稱視為身份的一部分
  - 首頁 hero pills 會顯示本團實際運費、`面交取貨免運`，以及跟隨 round 設定的面交點名稱
  - 商品卡的累計文案已改成 `已有 x被預訂，共有 y`，公開進度條同步改用 `庫存` / `已被預訂`
  - 商品卡圖面標題已移到圖片垂直中段；round window 的 `開放中` badge 改為固定單行
  - admin dashboard / 訂單 / 待出貨 / 商品 / 開團 / 供應商頁已改成 server-first 載入，初始 round chrome 與 badge 不再卡在 client-side auth / hydration waterfall
  - admin 訂單與待出貨已改成 server-side pagination + URL filter state，不再一次抓整團所有訂單再在瀏覽器過濾
  - `通知到貨` 已改成背景派送，CSV 匯出改成分批串流，商品卡圖片改用 `next/image`
- 這輪 UI 重構沒有改動核心流程或 API 契約；如果出現行為問題，先假設是既有邏輯問題，不是這次設計重構刻意改規格。

## Email 模板（4 種）

| type | Subject | Content |
|------|---------|---------|
| `payment_confirmed` | 訂單確認 — ORD-XXX | 收貨人、明細、合計（含運費）、取貨方式 |
| `product_arrival` | 商品到貨通知 | 「您訂購的【商品名】已到達理貨中心」 |
| `shipment` | 出貨通知 — ORD-XXX | 「您的訂單已出貨 / 已備妥可至取貨點領取」+ 明細 |
| `order_cancelled` | 訂單取消通知 — ORD-XXX | 「您的訂單已被取消。」+ 取消原因 + 明細 |

Plain HTML templates，不需 React Email。
